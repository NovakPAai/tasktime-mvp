# Права доступа в Flow Universe

## Роли в системе

В системе существует два уровня ролей:

- **Системные роли** — назначаются пользователю при создании учётной записи или администратором. Определяют, что пользователь может делать во всей системе. Роли хранятся в таблице `UserSystemRole` (multi-role: пользователь может иметь несколько ролей одновременно).
- **Роли в проектах** — назначаются пользователю внутри конкретного проекта. Хранятся в таблице `UserProjectRole`, расширяют или уточняют доступ в рамках проекта.

---

## Системные роли

| Роль | Описание |
|------|----------|
| `SUPER_ADMIN` | Суперадминистратор. Обходит все проверки прав. Единственный, кто может создавать пользователей, назначать роли в проектах, изменять системные настройки и удалять пользователей. |
| `ADMIN` | Администратор системы. Управляет пользователями, проектами, командами, спринтами, задачами. Может назначать системные роли (кроме `SUPER_ADMIN` и `ADMIN`). |
| `RELEASE_MANAGER` | Менеджер релизов. Управляет релизами: создаёт, редактирует, меняет статусы, добавляет/удаляет задачи. Не управляет пользователями, спринтами и командами. |
| `USER` | Базовая роль. Назначается каждому пользователю автоматически и не может быть снята. Работает с задачами, комментариями, учётом времени, AI-функциями. Доступ к проектам — через членство в проекте (`UserProjectRole`). |
| `AUDITOR` | Аудитор. Читает системную статистику, логи активности и отчёты. Не создаёт и не изменяет данные. |

> **Важно:** Роли в системе аддитивные. Пользователь может иметь несколько ролей одновременно (например `USER` + `RELEASE_MANAGER`). `SUPER_ADMIN` обходит все проверки вне зависимости от других ролей.

> **Доступ к проектам для USER:** Пользователи с ролью `USER` (без `ADMIN`/`AUDITOR`/`RELEASE_MANAGER`) видят только те проекты и задачи, в которых они числятся участниками через `UserProjectRole`.

## Роли в проектах

Помимо системных ролей, пользователю можно назначить роль внутри конкретного проекта. Назначение выполняется только через `SUPER_ADMIN` в `/api/admin/users/:id/roles`.

| Роль | Описание |
|------|----------|
| `ADMIN` (проект) | Полный контроль над проектом: все действия с задачами, участниками, настройками. |
| `MANAGER` (проект) | Управление спринтами, назначение задач, работа с релизами. |
| `USER` (проект) | Работа с задачами: создание, редактирование, логирование времени, комментарии. |
| `VIEWER` (проект) | Только чтение данных проекта. |

> **Текущая реализация:** проектные роли используются для контроля доступа `USER` к проектам/задачам (`requireIssueAccess`, `requireProjectRole`). Для пользователей с системной ролью `ADMIN`, `RELEASE_MANAGER` или `AUDITOR` членство в проекте не требуется — они имеют глобальный read-доступ ко всем проектам.

## Особенности реализации

- **SUPER_ADMIN обходит все проверки.** Middleware `isSuperAdmin()` пропускает суперадминистратора без проверки роли.
- **Multi-role RBAC.** Роли хранятся в junction-таблице `UserSystemRole`. Пользователь может иметь несколько системных ролей; `hasAnySystemRole()` проверяет наличие хотя бы одной из требуемых.
- **Глобальный read-доступ к проектам.** Роли `ADMIN`, `RELEASE_MANAGER`, `AUDITOR` и `SUPER_ADMIN` имеют доступ ко всем проектам без явного членства. Обычный `USER` видит только проекты, где есть запись в `UserProjectRole`.
- **Сервисный уровень.** Некоторые проверки выполняются в сервисах, а не в middleware: редактирование/удаление комментария (автор или ADMIN), просмотр времени других пользователей (только ADMIN).
- **USER — базовая, неснимаемая роль.** При попытке снять роль `USER` сервис возвращает ошибку. Все остальные роли опциональны.
- **Аудит-лог.** Все мутации записываются в таблицу `AuditLog` с указанием пользователя, действия, сущности, IP-адреса и User-Agent. Чтение не логируется.
