Зачем это нужно?
Раньше нужно было открыть систему → найти задачу → скопировать описание → вставить в чат с AI → вернуться → вручную обновить статус. Теперь AI сам читает задачу из системы, работает по ней и записывает результат обратно.
Как это работает

Архитектура

Claude Code / Cursor (твой ноутбук) │ │ stdio (JSON-RPC 2.0, защищённый канал) ▼ flow-universe MCP сервер (tsx process, локально) │ DATABASE_URL из .env.production │ │ TCP через SSH-туннель ▼ localhost:5435 ←─── ssh -L 5435:localhost:5432 root@72.56.7.218 -N │ ▼ PostgreSQL на 72.56.7.218 └─ пользователь: flowuniverse_mcp ├─ SELECT — все таблицы ├─ INSERT — issues, comments, time_logs, ai_sessions, audit_logs ├─ UPDATE — issues (только статус и AI-флаги) └─ DELETE/DDL — запрещено

Туннель нужен потому что Postgres на сервере слушает только localhost — снаружи недоступен. SSH-туннель пробрасывает его к тебе на ноутбук безопасно, внутри зашифрованного канала.

Настройка

Запуск за 5 шагов

1

Создать техпользователя в PostgreSQL (один раз)

На каждом сервере (staging и production) выполнить SQL-скрипт. Сначала придумать пароль для flowuniverse_mcp и вставить вместо CHANGE_ME.

bash # Подключиться к серверу
ssh root@72.56.7.218

# Создать пользователя (заменить CHANGE_ME на реальный пароль)
sed 's/CHANGE_ME_STRONG_PASSWORD/МОЙ_ПАРОЛЬ/' \
  /opt/tasktime/backend/scripts/mcp-db-user.sql \
  | docker exec -i tasktime-postgres-1 psql -U postgres -d tasktime
⚠️
Пароль сохранить в надёжном месте (1Password / командный Vault). В репо не коммитить.
2

Создать локальные env-файлы

Скопировать примеры и вставить реальный пароль от flowuniverse_mcp.

bash cd /путь/к/tasktime-mvp/backend

# Для staging
cp .env.staging.example .env.staging
# Открыть .env.staging, вписать пароль в DATABASE_URL

# Для production
cp .env.production.example .env.production
# Открыть .env.production, вписать пароль в DATABASE_URL

Строка которую нужно заменить:

DATABASE_URL="postgresql://flowuniverse_mcp:CHANGE_ME@localhost:5435/tasktime?schema=public"
3

Установить зависимости backend

bash cd backend && npm install
4

Открыть SSH-туннель (держать терминал открытым)

bash # Staging (пробросит localhost:5434 → 5.129.242.171:5432)
ssh -L 5434:localhost:5432 root@5.129.242.171 -N

# Production (пробросит localhost:5435 → 72.56.7.218:5432)
ssh -L 5435:localhost:5432 root@72.56.7.218 -N

Флаг -N означает «только туннель, без команды». Терминал будет висеть — это нормально.

5

Запустить MCP Inspector и проверить

bash cd backend && npx @modelcontextprotocol/inspector npm run mcp

Откроется http://localhost:6274. В поле:

  • Command: npm
  • Arguments: run mcp

Нажать Connect → List Tools → должны появиться 14 инструментов.

Проверить: get_issue { "key": "TTMP-1" } → задача из системы.

💡
Для production нужно передать переменную окружения:
MCP_ENV=production npm run mcp
Функционал

14 инструментов

📋 Задачи
get_issue Полная карточка задачи по ключу (TTMP-95) или UUID READ
list_issues Список задач с фильтрами: проект, спринт, статус, AI-eligible READ
update_status Изменить статус задачи (OPEN → IN_PROGRESS → REVIEW → DONE) WRITE
create_subtask Создать подзадачу под родительской задачей — реальная декомпозиция WRITE
🏃 Спринты
get_sprint_context Активный спринт: статистика, разбивка по AI-eligible задачам READ
get_backlog Задачи в бэклоге (не привязаны к спринту) READ
⏱️ Время и сессии
log_time Залогировать часы на задачу (source=AGENT, видно в отчётах) WRITE
register_ai_session Записать метрики сессии: токены, стоимость, распределение по задачам WRITE
get_time_summary Разбивка времени на задаче: человек vs агент READ
💬 Комментарии
add_comment Добавить комментарий к задаче от имени Flow Universe Agent WRITE
get_comments Получить последние комментарии к задаче READ
🤖 Агентский цикл
get_eligible_issues Очередь AI-eligible задач ещё не взятых в работу AGENT
claim_issue Атомарно взять задачу: IN_PROGRESS + AGENT (без гонки состояний) AGENT
complete_issue Завершить: статус → REVIEW, добавить саммари комментарий AGENT
fail_issue Эскалация: не смог выполнить, вернуть в OPEN с причиной AGENT
Примеры

Как это выглядит в диалоге

Сценарий 1 — Начало рабочего дня

Что у меня в активном спринте? Какие задачи можно взять в работу?
get_sprint_context TTMP list_issues project=TTMP sprint=active aiEligible=true

Активный спринт «Sprint 12» — 18 задач, из них 5 AI-eligible:
• TTMP-152 — GitHub webhook: Smart Commits (OPEN, MEDIUM)
• TTMP-148 — Экспорт задач в CSV (OPEN, LOW)
• TTMP-143 — Уведомления Telegram (OPEN, HIGH)
...
Возьми TTMP-152
get_issue TTMP-152 claim_issue TTMP-152

Взял TTMP-152 в работу. Статус → IN_PROGRESS, задача закреплена за агентом.

Сценарий 2 — Декомпозиция задачи

Декомпозируй TTMP-152 на подзадачи
get_issue TTMP-152

Создаю подзадачи по описанию...

create_subtask TTMP-152 "Парсинг issue key из commit message" create_subtask TTMP-152 "HMAC-верификация X-Hub-Signature-256" create_subtask TTMP-152 "Маппинг событий GitHub → статусы"

Создал 3 подзадачи: TTMP-153, TTMP-154, TTMP-155. Можешь видеть их в системе.

Сценарий 3 — Завершение и учёт времени

Готово. Залогируй сессию: 1.5 часа, $0.80
register_ai_session model=claude-sonnet-4-6 issues=[{TTMP-152,1.0}] costUSD=0.80 complete_issue TTMP-152 "Реализован GitHub webhook..."

Залогировал: 1.5ч, $0.80. Задача TTMP-152 → REVIEW.
Добавил комментарий с саммари. Ждёт твоего аппрува.
Безопасность

Что агент может и не может

Операция Разрешено Комментарий
SELECT * FROM issues ✓ Да Читать все таблицы
INSERT INTO issues ✓ Да Создавать подзадачи
UPDATE issues SET status ✓ Да Только поля задач
DELETE FROM issues ✗ Нет Явно запрещено
UPDATE users SET password ✗ Нет UPDATE выдан только на issues
DROP TABLE / ALTER TABLE ✗ Нет DDL недоступен, не owner
TRUNCATE ✗ Нет Явно отозвано
🛡️
Максимум что может случиться — агент создаст лишние подзадачи или поставит неправильный статус. Это исправляется руками за минуту. Структура базы, пользователи и настройки системы неприкосновенны.
