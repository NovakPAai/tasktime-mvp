# Роль: MANAGER / RELEASE_MANAGER — Руководство

> Для: PM, тимлидов, релиз-менеджеров, руководителей проектов
> Обновлено: 2026-04-19

В TaskTime две связанных роли с правами на управление релизами:
- **Проектный `MANAGER`** — полный контроль в рамках одного проекта.
- **Системный `RELEASE_MANAGER`** — глобальный доступ к релизам (сквозь проекты), CRUD типов и шаблонов КТ, bulk-apply на несколько релизов. Типично назначается одному сотруднику на контур организации.

## Что может MANAGER / RELEASE_MANAGER?

| Функция | MANAGER | RELEASE_MANAGER |
|---------|---------|-----------------|
| Создание проектов | ✓ | — |
| Создание и управление задачами (все задачи проекта) | ✓ | — (вне проекта только read через global-role) |
| Назначение исполнителей | ✓ | — |
| Создание/запуск/закрытие спринтов | ✓ | — |
| Управление релизами проекта | ✓ | ✓ (в любом проекте) |
| Управление INTEGRATION-релизами (кросс-проектные) | — | ✓ |
| Применение шаблонов КТ к релизу | ✓ (в своих проектах) | ✓ |
| **Bulk-apply** шаблона на несколько релизов сразу | — | ✓ (FR-21) |
| CRUD типов / шаблонов КТ | — | ✓ |
| Sync-instances типов КТ (FR-15) | — | ✓ |
| Ручной пересчёт КТ на релизе | ✓ | ✓ |
| Backfill burndown-снапшота | — | — (только SUPER_ADMIN / ADMIN, SEC-8) |
| Управление командами | ✓ | — |
| Просмотр отчётов и аудит-лога | ✓ | ✓ (audit через `AUDITOR` flag отдельно) |
| Bulk-операции над задачами | ✓ | — |
| Управление пользователями | — (только ADMIN) | — |
| Системный мониторинг | — | — |

## Типичный рабочий день PM / релиз-менеджера

1. **Утро:** Открыть Дашборд → проверить активный спринт + бейдж «Мои нарушения» в TopBar (если есть личные задачи с нарушенными КТ — поправить в первую очередь).
2. **Стендап:** Доска → фильтр «Исполнитель: по одному» → обсудить статусы; индикатор ⚠️ на карточке задачи подскажет, если задача нарушает КТ активного релиза.
3. **Планирование спринта:** Спринты → создать спринт → назначить задачи из бэклога.
4. **Назначение задач:** Открыть задачу → поле «Исполнитель» → выбрать.
5. **Работа с релизом (релиз-менеджер):**
   - Открыть релиз → вкладка **«Контрольные точки»** → проверить уровень риска (`LOW` / `MEDIUM` / `HIGH` / `CRITICAL`).
   - Если релиз новый: кнопка **«Применить шаблон»** → выбрать подходящий набор КТ (напр. «Release checklist v2») → preview покажет, какие задачи пройдут / нарушат каждую КТ → подтвердить.
   - Переключить на **«Матрица»** для детальной картины: строки = задачи, колонки = КТ. Иконки 🟢/🔴/🟡/— с tooltip-reason. «Экспорт CSV» — для внешнего отчёта.
   - Вкладка **«Диаграмма сгорания»** → посмотреть отставание actual от ideal. Для просроченных релизов ideal-линия flat — сигнал «мы за дедлайном».
   - При изменении состава релиза или движении задач — КТ пересчитываются автоматически (cron каждые 10 мин + event-хуки). Ручной «Пересчитать» — для отладки.
6. **Отчёт:** Администрирование → Отчёты → выбрать проект → экспортировать данные. Аудит событий нарушений КТ доступен для роли AUDITOR через `/admin/checkpoint-audit`.

## Типичные задачи релиз-менеджера

- **Настроить типы КТ под свои процессы.** `/admin/release-checkpoint-types` → создать тип с критерием (статус / исполнитель / срок / подзадачи / кастомное поле / без блокирующих связей), указать вес (`CRITICAL`/`HIGH`/`MEDIUM`/`LOW`) и смещение в днях от `plannedDate`.
- **Собрать шаблон.** `/admin/release-checkpoint-templates` → новый шаблон → добавить нужные типы → клон-кнопка для A/B-вариантов.
- **Применить шаблон на все ATOMIC-релизы квартала.** В `BulkApplyTemplateModal` выбрать релизы → ответ `207 Multi-Status` покажет, на какие применилось (200), а какие отклонены (403/409) с причиной.
- **Изменить критерии уже применённой КТ.** Править тип → нажать **«Синхронизировать инстансы»** → система обновит `criteriaSnapshot` / `offsetDaysSnapshot` на всех существующих КТ-экземплярах (FR-15).
- **Отправить уведомление при смене состояния.** В типе КТ указать `webhookUrl` (Slack/MS Teams incoming webhook). При переходе OK → VIOLATED или VIOLATED → OK будет POST с debounce `minStableSeconds` (FR-17).
- **Провести ревью.** `/admin/checkpoint-audit` (нужна роль `AUDITOR`) → фильтры по периоду / проекту / типу / релизу → Экспорт CSV.
