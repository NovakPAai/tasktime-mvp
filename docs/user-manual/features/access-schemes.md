# Схемы доступа — Руководство пользователя

> Функция: Проектные схемы ролей и прав (RBAC) + группы пользователей
> Доступ: Настройка схем — **SUPER_ADMIN**; Назначение ролей / управление группами — ADMIN
> Путь: Меню → Администрирование → **Схемы доступа** (`/admin/role-schemes`), **Группы** (`/admin/user-groups`); Профиль → «Безопасность» (self-view)
> Обновлено: 2026-04-18 (TTSEC-2)

---

## Что такое схема доступа?

**Схема доступа** — это набор ролей и матрица их прав, которую можно **привязать к проекту**. Разные проекты могут использовать разные схемы: одна команда получает один набор правил, другая — другой.

Примеры использования:
- Команда безопасности хочет запретить даже менеджерам удалять задачи — создают схему «InfoSec» без `ISSUES_DELETE` у MANAGER.
- Для R&D-проекта нужна новая роль `DEVOPS` с доступом только к релизам — добавляют её в схему «R&D».
- Для онбординга создают схему «Stagiaire», где участники видят всё, но ничего не могут менять.

---

## Ключевые понятия

| Термин | Описание |
|--------|----------|
| **Схема (Scheme)** | Именованный набор ролей + матрица прав |
| **Дефолтная схема** | Одна на весь экземпляр. Применяется к проектам без явной привязки. Нельзя удалить, нельзя снять флаг у последней |
| **Роль в схеме** | `ADMIN` / `MANAGER` / `USER` / `VIEWER` (системные) + любые кастомные (`DEVOPS`, `AUDITOR`, ...) |
| **Ключ роли (key)** | Идентификатор внутри схемы. Ключи системных ролей зарезервированы |
| **Системная роль** | Роль с `isSystem=true`. Добавляется автоматически в каждую новую схему. Нельзя удалить |
| **Кастомная роль** | Роль с `isSystem=false`, созданная пользователем. Можно удалить, если нет назначенных пользователей |
| **Привязка проект ↔ схема** | Один проект — одна схема. Без привязки используется дефолтная |

## Страница «Схемы доступа»

Список всех схем с индикатором «по умолчанию». Действия:

- **+ Новая схема** — создать. Автоматически копируются 4 системных роли + матрица прав из текущей дефолтной схемы.
- **Переименовать** — только `name` / `description` / `isDefault`.
- **Сделать дефолтной** — флаг `isDefault=true` сбрасывается у текущей дефолтной и ставится у выбранной атомарно. Кэш схем проектов инвалидируется.
- **Удалить** — только если к схеме не привязан ни один проект и это **не** дефолтная.

---

## Страница детализации схемы

Открывается кликом по имени. Две вкладки:

### Вкладка «Роли»

Список системных (`ADMIN`/`MANAGER`/`USER`/`VIEWER`) и кастомных ролей.

- **+ Новая роль** — имя, ключ (A-Z_), цвет. Ключи системных ролей зарезервированы.
- **Настроить права** — открывает drawer с матрицей permissions (см. ниже).
- **Редактировать** — `name`, `description`, `color`. Ключ не меняется.
- **Удалить** — доступно только для кастомной роли и только если к ней не привязан ни один участник (ошибка `ROLE_IN_USE: роль назначена N пользователям`). Системные роли удалить нельзя.

### Вкладка «Проекты»

Список проектов, привязанных к этой схеме + добавление новых.

- **Привязать проект** — если проект уже привязан к другой схеме, его назначения пользователей автоматически ремапятся по ключу роли. Если у проекта есть роль с ключом, отсутствующим в целевой схеме, операция отклоняется с 400 и перечнем несовместимых ключей.
- **Отвязать проект** — проект откатывается на дефолтную схему, назначения ремапятся туда же.

### TTSEC-2: гранулярные permissions

В Phase 1/2/3 было введено разбиение крупно-гранулярных прав на CRUD:

- **`SPRINTS_MANAGE` / `RELEASES_MANAGE`** убраны из UI матрицы (остаются в enum как deprecated — PostgreSQL не поддерживает `DROP VALUE`). Вместо них — `*_CREATE` / `*_EDIT` / `*_DELETE`. Роль можно настроить под «создавать, но не удалять» спринты.
- **`COMMENTS_DELETE_OTHERS`** и **`TIME_LOGS_DELETE_OTHERS`** — модерация чужих записей. Автор всегда может удалять/редактировать **свои** записи без отдельного permission. Для удаления чужих нужен `*_DELETE_OTHERS` ИЛИ `*_MANAGE` (OR-список).
- **`USER_GROUP_VIEW` / `USER_GROUP_MANAGE`** — system-level permissions для админки групп. По умолчанию выдаются только ADMIN-у. Фронт использует helper `canViewUserGroups(user)` из `frontend/src/lib/roles.ts` — сейчас это прокси на системную роль `ADMIN`, и это единственная точка, где в Phase 4+ можно будет заменить логику на реальный `hasSystemPermission(user, 'USER_GROUP_VIEW')`, когда бэкенд начнёт экспозить per-user permissions.

**Backfill при миграции на Phase 1**: каждая роль, у которой был granted `SPRINTS_MANAGE`, получила `SPRINTS_CREATE + EDIT + DELETE`. Аналогично для `RELEASES_MANAGE`. Роли с `COMMENTS_MANAGE` / `TIME_LOGS_MANAGE` дополнительно получили `*_DELETE_OTHERS`. Эффективный уровень доступа сохранён.

## Как это работает в рантайме

При каждом запросе к защищённому эндпоинту:

1. `requireProjectPermission(..., 'ISSUES_EDIT')` middleware проверяет, есть ли у пользователя право.
2. `SUPER_ADMIN` — всегда проходит.
3. Для read-permission (`*_VIEW`) — глобальные роли `ADMIN`/`RELEASE_MANAGER`/`AUDITOR` проходят без проверки членства (opt-out через `allowGlobalRead: false`).
4. Иначе — резолвим схему проекта (привязка → иначе дефолтная), берём роль пользователя, ищем permission в её матрице.
5. Результат «разрешено» кэшируется на 60с в Redis. «Запрещено» не кэшируется — новая роль даёт доступ сразу.

**Инвалидация кэша** при назначении/удалении роли пользователя, смене схемы проекта, изменении матрицы или состава ролей.

## Ограничения и инварианты

- Один пользователь — одна роль в одном проекте (`@@unique([userId, projectId])`). При повторном назначении той же роли — идемпотентный ответ; другой роли — 409.
- `roleId` и `schemeId` в `UserProjectRole` хранятся вместе (composite FK + CHECK constraint). Роль нельзя удалить, если на неё есть назначения (`ON DELETE RESTRICT`).
- Всегда ровно одна дефолтная схема (enforced в сервисе; seed атомарно восстанавливает инвариант).
- Ключи кастомных ролей не могут совпадать с системными (`ADMIN`/`MANAGER`/`USER`/`VIEWER`).
- Backfill существующих `UserProjectRole.roleId` при миграции резолвит роль по активной схеме каждого проекта, не сбрасывает всех на дефолтную.

---

## Миграция со старой модели

Старые назначения ролей с `role='ADMIN/MANAGER/USER/VIEWER'` и `roleId=NULL` автоматически заполняются в `seed.ts` при первом запуске после мёрджа: для каждого проекта определяется активная схема, подбирается роль по legacy-ключу, сохраняются `roleId` и `schemeId`.

Если в базе существовало несколько ролей у одного пользователя в одном проекте, миграция `20260418000000_user_project_role_single` оставляет самую привилегированную (ADMIN > MANAGER > USER > VIEWER) и удаляет остальные **до** применения нового unique-индекса.

## Группы пользователей (TTSEC-2)

**Группа** — уровень абстракции между пользователем и проектной ролью. Вместо того чтобы раздавать роли каждому пользователю поштучно, админ:
1. Создаёт группу (например, «Frontend Team»).
2. Добавляет 12 участников одной операцией.
3. Выдаёт группе роль `Developer` в пяти проектах — все 12 получают права разом.
4. При увольнении одного — убирает его из группы, доступы снимаются централизованно.

### Страница «Группы» (`/admin/user-groups`)

Таблица всех групп со счётчиками участников и проектных биндингов. Действия:
- **Создать** — `name` (глобально уникальное) + optional `description`. Дубликаты имени — 409.
- **Настроить** (клик по имени) — детализация с двумя табами:
  - **Участники** — список user-ов с датой добавления и кем добавлен. Batch add (multi-select с server-side поиском для больших инсталляций). Удаление одной кнопкой.
  - **Проектные роли** — один биндинг на проект (`@@unique([groupId, projectId])`). При добавлении выбирается проект → автоматически подгружается его активная схема → в селекте ролей только роли из этой схемы (кросс-схемный binding отклоняется 400 на бэкенде).
- **Переименовать** — `name` / `description`.
- **Удалить** — требует `?confirm=true`. Модалка предварительно показывает impact: список затронутых участников + проектных биндингов. Без подтверждения — 412 + impact в ответе.

Все мутации логируются в `AuditLog`: `user_group.created/renamed/updated/deleted/members_changed`, `project_group_role.granted/revoked`.

### Эффективная роль пользователя в проекте

Когда пользователь открывает защищённую страницу проекта, бэкенд вычисляет **одну** эффективную роль. Алгоритм (`backend/src/shared/middleware/rbac.ts::computeEffectiveRole`):

1. **Собираются кандидаты** — уникальные роли, к которым у юзера есть путь доступа:
   - Прямая роль из `UserProjectRole` (если есть).
   - Роли через каждую группу, в которой состоит юзер и которая привязана к этому проекту.
   - Если одна и та же роль достижима несколькими путями — это **один** кандидат (дедуп по `roleId`); пути запоминаются в `sourceGroups[]`.
2. **Выбирается кандидат** с максимальным числом granted permissions. Тайбрейкер — `roleId` в лексикографическом порядке (детерминированно).
3. **Присваивается `source` выбранному кандидату**: `DIRECT`, если среди путей к этой роли есть прямое назначение; иначе `GROUP`. Это просто маркер для UI — он **не** влияет на выбор в п.2 и не переопределяет более привилегированную групповую роль. Если `USER` назначен напрямую, а через группу `ADMIN` — `ADMIN` выигрывает (max permissions) с `source='GROUP'`.
4. **`sourceGroups[]`** — полный список групп, через которые достижима выбранная роль (независимо от `source`). Используется для отображения «прямое назначение · также через Frontend Team».

Результат (set granted permissions) кэшируется в Redis `rbac:effective:{projectId}:{userId}` на 60с.

### Legacy-группы (миграция Phase 1)

Миграция `20260421000001_ttsec2_groups_and_backfill` **копирует** каждое прямое назначение в group-based: для каждой уникальной пары `(project, role)` в таблице `user_project_roles` создаётся группа вида `Legacy: {project.key} — {role.name}` с биндингом на этот проект+роль; пользователи с соответствующим прямым назначением автоматически добавляются в членство группы.

**Важно:** старые строки `UserProjectRole` **не удаляются** миграцией. После Phase 1 у юзера остаётся и прямая запись, и членство в Legacy-группе. Оба пути ведут к одной и той же `roleId` → `computeEffectiveRole` видит одного кандидата (дедуп), и `source='DIRECT'`. Эффективный уровень доступа идентичен до и после миграции.

Причины, по которым прямые строки сохранены:
- **Break-glass**: если что-то сломается в групповом резолвере, прямые записи продолжают работать.
- **Обратимость**: rollback миграции не требует восстановления удалённых прямых записей.
- **Cutover — отдельный этап**: запрет на НОВЫЕ прямые назначения включается флагом `FEATURES_DIRECT_ROLES_DISABLED` (см. ниже), а миграция старых строк в группы — опциональная уборка после успешной валидации.

Префикс `Legacy:` позволяет фильтровать такие группы в UI, а при желании — массово удалить вместе с соответствующими прямыми записями отдельным cleanup-ом.

## Вкладка «Безопасность» в профиле

Путь: Настройки → карточка «Безопасность» (отдельный блок на `/settings`).

Read-only view с тремя блоками:
1. **Мои группы** — список групп, в которых состоит пользователь, со счётчиком членов каждой.
2. **Мои роли в проектах** — таблица `Проект | Роль | Источник`. Роль — тег с tooltip на permissions. Источник — «Прямое назначение» или «Через группу» + список групп (если через группы).
3. **Экспорт CSV** — скачивает таблицу (RFC 4180 escaping + UTF-8 BOM для корректной кириллицы в Excel).

API:
- `GET /api/users/me/security` — self-view (любой аутентифицированный юзер).
- `GET /api/admin/users/:id/security` — admin-view для конкретного юзера.

## Cutover на группы (Phase 4)

После того как вся команда мигрировала на групповое управление ролями, прямые назначения через `POST /api/admin/users/:id/project-roles` можно отключить на уровне среды.

**Feature flag** (backend): `FEATURES_DIRECT_ROLES_DISABLED=true` в env. Когда включён:
- `admin.assignProjectRole` отклоняется с 403 + сообщение, указывающее на `/admin/user-groups` как альтернативу.
- Существующие `UserProjectRole` с `source='DIRECT'` **остаются функциональны** — их можно оставить как break-glass escape hatch либо мигрировать в группы отдельно.

**Рекомендованный порядок включения:**
1. Staging: `FEATURES_DIRECT_ROLES_DISABLED=true`, убедиться что все сценарии работают через группы.
2. Prod: включить env-variable (флаг читается при каждом запросе, redeploy не нужен).

**Откат:** просто снять env-variable.
