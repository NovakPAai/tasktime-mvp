# Контрольные точки релиза — Руководство пользователя

> Обновлено: 2026-04-19 (TTMP-160)

---

## Что такое контрольные точки?

**Контрольная точка (КТ)** — это заранее определённая дата, к которой задачи в релизе должны соответствовать заданным критериям (например, «все баги переведены в DONE за 3 дня до релиза» или «все эпики имеют назначенного исполнителя»). Нарушения агрегируются в **уровень риска** релиза: `LOW` / `MEDIUM` / `HIGH` / `CRITICAL`.

Модуль помогает релиз-менеджеру:
- видеть заранее, какие задачи не успевают попасть в релиз;
- применять к релизу готовый набор КТ одним действием (шаблон);
- получать агрегированную сводку риска и детализацию по каждой КТ;
- экспортировать нарушения в CSV для внешних отчётов.

---

## Где искать

- **Карточка релиза (глобальная страница «Релизы»)** — вкладка «Контрольные точки». Два вида:
  - **Список** (по умолчанию) — одна КТ = одна карточка со «светофором» и списком нарушений.
  - **Матрица** — строки = задачи, столбцы = КТ, цветные иконки состояния (🟢 passed / 🔴 violated / 🟡 pending / — n/a). Экспорт в CSV.
- **Карточка релиза** — вкладка «Диаграмма сгорания» — см. [release-burndown.md](./release-burndown.md).
- **Задача** (`IssueDetailPage`) — секция «Контрольные точки» показывает все КТ активных релизов, в которые входит задача.
- **Доска** (`BoardPage`) — индикатор ⚠️ на карточке задачи, если у неё хотя бы одна открытая violation.
- **Дашборд / TopBar** — бейдж «Мои нарушения» со счётчиком; по клику — фильтрованный список.

### 👨‍💻 Разработчик (`USER`)

**Что видно:** только задачи своих проектов; на задаче — блок «Контрольные точки» со списком активных КТ релизов, в которые входит задача, и причиной нарушения.

**Что можно:**

| Действие | Где | Комментарий |
|----------|-----|-------------|
| Посмотреть, какие КТ «горят» на мне | TopBar → бейдж «Мои нарушения» | Счётчик из `/api/my-checkpoint-violations/count`. Клик → список с группировкой по релизу (FR-12). |
| Увидеть ⚠️ на карточке задачи | Доска | Индикатор на задаче с хотя бы одной open violation. |
| Посмотреть детали нарушения | Задача → секция «Контрольные точки» | Показывает reason, deadline, релиз. |
| Исправить нарушение | Обычный workflow — перевести статус в `DONE`, заполнить поле, закрыть подзадачи | Recompute произойдёт автоматически (event-хук + cron). |

**Чего нельзя:** применять / удалять КТ, менять критерии, видеть релизы чужих проектов.

---

### 🧭 Релиз-менеджер (`RELEASE_MANAGER` или проектный `MANAGER`)

**Что видно:** все свои релизы + все КТ на них + полный UI управления (шаблоны, матрица, burndown).

**Что можно:**

| Действие | Где | Комментарий |
|----------|-----|-------------|
| Применить шаблон к релизу | Релиз → вкладка «Контрольные точки» → «Применить шаблон» | Preview-модалка покажет, какие задачи пройдут / нарушат каждую КТ до подтверждения (FR-14). |
| Добавить отдельные КТ вручную | Релиз → «Контрольные точки» → «+ добавить тип» | Выбор одного или нескольких типов без шаблона. |
| Удалить КТ с релиза | Иконка корзины на карточке КТ | |
| Ручной пересчёт | Кнопка «Пересчитать» в шапке вкладки | Обычно не нужен — cron каждые 10 мин + event-хуки покрывают. Используется для отладки. |
| Просмотреть матрицу «Задачи × КТ» | Вкладка «Контрольные точки» → переключатель «Матрица» | 🟢 passed / 🔴 violated / 🟡 pending / — n/a. Tooltip с reason. |
| Экспортировать матрицу в CSV | Кнопка «Экспорт CSV» в шапке матрицы | UTF-8 BOM + CRLF — читается в Excel с кириллицей. |
| Посмотреть burndown | Вкладка «Диаграмма сгорания» | actual + ideal + метрики (задачи / часы / нарушения). |
| Увидеть риск-скоринг релиза | Плашка «Риск» в шапке релиза | `LOW` / `MEDIUM` / `HIGH` / `CRITICAL` (FR-25). |

**Дополнительно для системного `RELEASE_MANAGER` (сквозь проекты):**

| Действие | Где | Комментарий |
|----------|-----|-------------|
| Создавать / редактировать **типы** КТ | `/admin/release-checkpoint-types` | CRUD + drag-drop порядка критериев. |
| Создавать / редактировать **шаблоны** КТ | `/admin/release-checkpoint-templates` | Клон, reorder. |
| **Sync-instances** типа | В админ-UI типа → «Синхронизировать инстансы» | Обновляет `criteriaSnapshot` / `offsetDaysSnapshot` на всех существующих экземплярах (FR-15). |
| **Bulk-apply** шаблона на несколько релизов | `BulkApplyTemplateModal` | Ответ `207 Multi-Status`: per-release `200 OK` или `403/409` с причиной (FR-21). |

**Чего нельзя:** backfill burndown-снапшотов (только SUPER_ADMIN / ADMIN, SEC-8) — чтобы нельзя было задним числом «переписать историю».

### 📊 PM / продуктовый менеджер

**Что видно:** вкладки «Контрольные точки» и «Диаграмма сгорания» на любом релизе своих проектов (read-only, если нет `RELEASES_EDIT`).

**Что можно:**

| Действие | Комментарий |
|----------|-------------|
| Смотреть состояние КТ и уровень риска | — |
| Смотреть матрицу и burndown | — |
| Применять шаблоны | только если имеет permission `RELEASES_EDIT` или системную роль `RELEASE_MANAGER` |

**Чего нельзя:** создавать типы / шаблоны (только `RELEASE_MANAGER` / `ADMIN`), backfill (только `ADMIN`/`SUPER_ADMIN`).

---

### 🔍 Аудитор (`AUDITOR`)

**Что видно:** read-only доступ ко всем релизам всех проектов, КТ, событиям нарушений и burndown-истории.

**Что можно:**

| Действие | Где | Комментарий |
|----------|-----|-------------|
| Смотреть все КТ всех релизов | Любая страница релиза | Read-only (кнопки управления скрыты). |
| Открыть страницу аудита КТ | `/admin/checkpoint-audit` | Все события нарушений (open + resolve) со всеми фильтрами. |
| Фильтровать аудит | По периоду / проекту / типу / релизу | — |
| Экспорт аудита в CSV | Кнопка «Экспорт CSV» | UTF-8 BOM + CRLF; поля — `issueKey`, `issueTitle`, `status` (без описаний / assignee / кастомных полей, SEC-9). |

**Чего нельзя:** мутировать что-либо. Операции CRUD, apply-template, recompute, backfill — все 403.

---

### ⚙️ Администратор (`SUPER_ADMIN` / `ADMIN`)

**Что видно:** всё, что видят выше + дополнительно системные операции.

**Что можно дополнительно:**

| Действие | Где | Комментарий |
|----------|-----|-------------|
| **Backfill** burndown-снапшота | Релиз → «Диаграмма сгорания» → «Создать снапшот» | Можно указать дату в теле `POST /backfill { date }`. SEC-8: доступен только роли `ADMIN`/`SUPER_ADMIN`. |
| Глобальный доступ к межпроектным INTEGRATION-релизам | — | `RELEASES_VIEW` и `RELEASES_EDIT` на любом релизе. |
| CRUD типов / шаблонов / bulk-apply | см. блок RELEASE_MANAGER выше | ADMIN делает то же самое, без ограничений по проектам. |
| Настройка cron-расписания | `.env` → `CHECKPOINTS_SCHEDULER_CRON`, `BURNDOWN_SNAPSHOT_CRON`, `BURNDOWN_RETENTION_CRON` | Требуется перезапуск backend-service. |
| Выключить scheduler | `.env` → `CHECKPOINTS_SCHEDULER_ENABLED=false` | Event-хуки продолжат работать; только cron выключится. |

## Типы и шаблоны (для `SUPER_ADMIN` / `ADMIN` / `RELEASE_MANAGER`)

- `/admin/release-checkpoint-types` — CRUD типов: название, цвет, вес (`CRITICAL` / `HIGH` / `MEDIUM` / `LOW`), смещение в днях от `plannedDate`, критерии (`STATUS_IN`, `DUE_BEFORE`, `ASSIGNEE_SET`, `CUSTOM_FIELD_VALUE`, `ALL_SUBTASKS_DONE`, `NO_BLOCKING_LINKS`).
- `/admin/release-checkpoint-templates` — CRUD шаблонов (именованный набор типов). Клон, bulk-apply на несколько релизов.
- **Sync-instances** — если у типа изменились критерии или offset, админ может синхронизировать уже созданные КТ на релизах одной операцией (FR-15).

## Состояния КТ

| Состояние | Когда | Светофор |
|-----------|-------|----------|
| **PENDING** без нарушений | `now < deadline`, нет задач, нарушающих критерии | ⚪ серый («Ожидание») |
| **PENDING + WARNING** | `now < deadline`, есть нарушения, до дедлайна ≤ `warningDays` дней | 🟠 жёлтый/оранжевый («Внимание») |
| **PENDING** вне warning-окна | `now < deadline`, есть нарушения, до дедлайна > `warningDays` дней | ⚪ серый (но в tooltip видны нарушения) |
| **OK** («Пройдено») | `now ≥ deadline` и нарушений нет. **Финальный успех.** | 🟢 зелёный |
| **VIOLATED** («Нарушено») | `now ≥ deadline` и остались задачи, не удовлетворяющие критериям. **Финальная неудача.** | 🔴 красный |

**Важно:** КТ **не показывается как «Пройдено» до наступления дедлайна**, даже если в данный момент все применимые задачи удовлетворяют критериям. Это намеренно — статус «Пройдено» становится финальным только после deadline (задачи ещё могут добавляться / переоткрываться).

`WARNING` — это не отдельное значение `CheckpointState` (поле `state` в БД), а UI-подсветка поверх `PENDING`, вычисляемая по флагу `isWarning`. В API `isWarning=true` приходит рядом со `state: "PENDING"`.

## Матрица «Задачи × КТ»

Вкладка «Контрольные точки» → переключатель «Список / Матрица»:
- **строки** — задачи релиза;
- **столбцы** — КТ;
- **ячейки**:
  - 🟢 **passed** — задача прошла критерии КТ;
  - 🔴 **violated** — задача нарушила КТ (tooltip с причиной);
  - 🟡 **pending** — дедлайн КТ ещё не наступил;
  - `—` **n/a** — задача не попадает под `issueTypes`-фильтр критерия (например, KT применима только к `BUG`, а задача — `TASK`).
- **Экспорт CSV** — кнопка «Экспорт CSV» в шапке матрицы. Формат: UTF-8 BOM + CRLF (читается в Excel с кириллицей без мусора).

## Риск-скоринг

`score = sum(weight_of_violated) / sum(weight_of_all)`

Веса: `CRITICAL=8`, `HIGH=4`, `MEDIUM=2`, `LOW=1`.

- `score = 0%` → `LOW`;
- `1-30%` → `MEDIUM`;
- `31-70%` → `HIGH`;
- `>70%` → `CRITICAL`.

Уровень отображается цветной плашкой «Риск» в шапке релиза (FR-25).

## Режим условия: Structured / TTQL / Combined (TTSRH-1, 2026-04-21)

С эпика TTSRH-1 у каждого **типа контрольной точки** появилось поле «Режим условия» — segmented control в форме админки `Администрирование → Контрольные точки → Типы`:

| Режим | Что настраивается | Когда использовать |
|-------|-------------------|--------------------|
| `STRUCTURED` (default) | список criteria[] (STATUS_IN, DUE_BEFORE, ASSIGNEE_SET, CUSTOM_FIELD_VALUE, ALL_SUBTASKS_DONE, NO_BLOCKING_LINKS) | простые детерминированные условия, которые нужны всем |
| `TTQL` | одно TTS-QL выражение в текстовом редакторе | комплексные условия (OR, сложные подзапросы, `violatedCheckpoints()`, кастомные поля) |
| `COMBINED` | и structured-critera, и TTQL (логическое AND) | нужна combo: «все подзадачи закрыты **И** custom-field проставлен» |

Backward-compat: все существующие КТ остаются в `STRUCTURED` после миграции — поведение идентично пред-TTSRH-1.

### TTS-QL редактор в форме типа КТ

В режимах `TTQL` / `COMBINED` появляется поле «TTS-QL условие» — тот же `JqlEditor` из страницы `/search`, но в контексте **КТ-variant**:
- Автокомплит расширен функциями `releasePlannedDate()`, `checkpointDeadline()` — доступны только здесь.
- `currentUser()` — **предупреждение** при save (не ошибка): CT эвалуируется scheduler'ом, в этом контексте `currentUser()` обычно не то, что имеется в виду.
- Валидация — та же, что в поиске: inline-ошибки в редакторе.

Полный reference TTS-QL — [jql.md](./jql.md).

### Preview-панель

Под формой — disclosure «Preview» — позволяет выбрать релиз и увидеть **без сохранения**, сколько задач окажется в каждой категории:

- **applicable** — задачи, к которым применимо условие (по `issueTypes`, попаданию в релиз).
- **passed** — удовлетворяют условию (`status = OK`).
- **violated** — нарушают (`state = WARNING` / `OVERDUE`).

Preview использует тот же path, что production scheduler — zero drift между preview и реальной эвалуацией.

### Конвертер Structured → TTQL

Кнопка «Сконвертировать structured-критерии в TTS-QL (draft)» — one-way генератор:

- Читает текущие `criteria[]` + фильтр `issueTypes`.
- Конвертирует каждый тип критерия в канонический TTS-QL (§5.12.9 в TZ).
- Переключает режим в `COMBINED` и вставляет draft в TTQL-editor.
- **Не** сохраняет автоматически — требует ручного ревью и подтверждения (**R21**).

Критерии `ALL_SUBTASKS_DONE` и `NO_BLOCKING_LINKS` пока не имеют прямого TTS-QL-эквивалента (требуют рекурсии) — конвертер вставляет `-- TODO:` placeholder для ручной доработки.

### Feature flag `FEATURES_CHECKPOINT_TTQL`

Под отдельным флагом:
- `false` (default): TTQL/COMBINED checkpoints эвалуируются как STRUCTURED (безопасный no-op). Preview показывает баннер «TTQL skipped by flag».
- `true` (после UAT): полная эвалуация TTQL.

Флаг позволяет мержить security-review ДО завершения UAT — fail-safe в production.

### Ошибки TTQL-эвалуации

Compile / runtime error / 5s timeout в TTQL → checkpoint переходит в `state = ERROR` + единичная violation с `criterionType = TTQL_ERROR`. Детерминированный `violationsHash` (R16, FR-31) — не меняется между тиками.

UI показывает баннер ERROR с кнопкой «Открыть тип и исправить» — отводит к форме типа КТ (и при необходимости — в конвертер / Structured-режим).
