Индексы PostgreSQL

Все 11 индексов должны присутствовать в pg_indexes
11 новых индексов созданы
Critical 
SELECT indexname, tablename
FROM pg_indexes
WHERE schemaname = 'public'
  AND indexname IN (
    'refresh_tokens_user_id_idx',
    'sprints_project_team_id_idx',
    'sprints_business_team_id_idx',
    'sprints_flow_team_id_idx',
    'comments_author_id_idx',
    'team_members_team_id_idx',
    'audit_logs_action_idx',
    'issues_created_at_idx',
    'issues_updated_at_idx',
    'issues_status_idx',
    'time_logs_log_date_idx'
  )
ORDER BY tablename;

EXPLAIN показывает Index Scan (не Seq Scan)
High

Проверить на реальном проекте, что запрос использует индекс.

-- Заменить <PROJECT_UUID> на реальный ID проекта
EXPLAIN ANALYZE
SELECT id, title, status, priority
FROM issues
WHERE project_id = '<PROJECT_UUID>'
  AND status = 'IN_PROGRESS'
ORDER BY created_at DESC
LIMIT 100;
Ожидаемый результат
→ Bitmap Index Scan on issues_project_id_status_idx или Index Scan using issues_created_at_idx
Если Seq Scan
Таблица слишком маленькая — PostgreSQL предпочитает seq scan на малых объёмах. Это нормально для dev/staging с небольшими данными. Индексы заработают при росте таблицы.

Pagination API

Endpoint'ы возвращают {data, meta} вместо массива
⚠
Получить JWT-токен перед тестами: POST /api/auth/login с ${ADMIN_EMAIL} / ${ADMIN_PASSWORD}
Подставить в переменную TOKEN во всех запросах ниже.
Авторизация — получить токен
Setup 
# Получить токен
TOKEN=$(curl -s -X POST http://${STAGING_HOST}/api/auth/login \
  -H "Content-Type: application/json" \
  -d '{"email":"${ADMIN_EMAIL}","password":"${ADMIN_PASSWORD}"}' \
  | jq -r .accessToken)

echo "Token: $TOKEN"

# Получить ID проекта
PROJECT_ID=$(curl -s http://${STAGING_HOST}/api/projects \
  -H "Authorization: Bearer $TOKEN" \
  | jq -r '.[0].id')

echo "Project: $PROJECT_ID"
GET /api/projects/:id/issues — paginated response
Critical 
# Первая страница, 10 записей
curl -s "http://${STAGING_HOST}/api/projects/$PROJECT_ID/issues?page=1&limit=10" \
  -H "Authorization: Bearer $TOKEN" | jq '{data_length: (.data | length), meta}'
Ожидаемый результат
{ "data_length": 10, "meta": { "page": 1, "limit": 10, "total": <N>, "totalPages": <K> } }
Если вернулся массив (без meta)
Фронтенд API-слой не обновлён или деплой не включил новые файлы. Проверить версию контейнера.

N+1 fix — GitLab webhook

Webhook должен генерировать ровно 3 запроса к БД, независимо от числа ключей
Тест webhook — 3 запроса к БД вместо 2N
Critical

Включить query logging на staging, отправить webhook с несколькими ключами и посчитать запросы.

# 1. Войти в контейнер backend и включить логи
docker exec -it tasktime-backend sh
# Установить в .env или запустить с переменной:
# DEBUG=prisma:query
# 2. Отправить тестовый webhook с 3 issue-ключами
curl -s -X POST http://${STAGING_HOST}/api/webhooks/gitlab \
  -H "Content-Type: application/json" \
  -d '{
    "object_kind": "push",
    "ref": "refs/heads/feature/DEMO-1",
    "commits": [
      {"message": "fix DEMO-2 and BACK-1 resolved"},
      {"message": "update DEMO-3"}
    ]
  }'
# 3. Проверить логи контейнера
docker logs tasktime-backend --tail 30 | grep -E "prisma:query|SELECT|UPDATE|INSERT"
Ожидаемый результат — ровно 3 запроса
Query 1: SELECT ... FROM "issues" WHERE ... IN (...) Query 2: UPDATE "issues" SET "status" = $1 WHERE "id" IN (...) Query 3: INSERT INTO "audit_logs" ...
Если запросов больше (кратно N ключам)
N+1 не исправлен — проверить, что задеплоился новый код gitlab.service.ts

Redis — кеш работает

Повторный запрос не должен генерировать запросы к БД
Ключи в Redis появляются после первого запроса
High 
# 1. Сделать первый запрос (заполнить кеш)
curl -s "http://${STAGING_HOST}/api/projects/$PROJECT_ID/issues?page=1&limit=10" \
  -H "Authorization: Bearer $TOKEN" > /dev/null

# 2. Проверить ключи в Redis
docker exec -it tasktime-redis redis-cli KEYS "issues:list:*"
Ожидаемый результат
1) "issues:list:<PROJECT_ID>:s=:t=:pr=:a=:sp=:fr=:to=:q=:pg=1:lm=10"
Второй запрос быстрее — обслуживается из кеша
High 
# Измерить время двух последовательных запросов
echo "=== Запрос 1 (промах кеша) ==="
time curl -s "http://${STAGING_HOST}/api/projects/$PROJECT_ID/issues?page=1&limit=50" \
  -H "Authorization: Bearer $TOKEN" > /dev/null

echo "=== Запрос 2 (попадание в кеш) ==="
time curl -s "http://${STAGING_HOST}/api/projects/$PROJECT_ID/issues?page=1&limit=50" \
  -H "Authorization: Bearer $TOKEN" > /dev/null
Ожидаемый результат
Запрос 1: ~50–200 ms (идёт в БД) Запрос 2: ~5–20 ms (Redis кеш)
TTL кеша — не более 30 секунд
Medium 
# Проверить TTL любого ключа
docker exec tasktime-redis redis-cli TTL "issues:list:<PROJECT_ID>:..."

# Или через scan + ttl
docker exec tasktime-redis redis-cli --scan --pattern "issues:*" \
  | head -1 | xargs -I{} docker exec tasktime-redis redis-cli TTL {}
Ожидаемый результат
Число от 1 до 30 (секунды до истечения)

Замер производительности

Убедиться, что p95 latency < 200ms (NFR)
p95 latency — 20 последовательных запросов
Medium 
# Снять 20 замеров и посмотреть распределение
for i in $(seq 1 20); do
  curl -o /dev/null -s -w "%{time_total}\n" \
    -H "Authorization: Bearer $TOKEN" \
    "http://${STAGING_HOST}/api/projects/$PROJECT_ID/issues?page=1&limit=50"
done | sort -n | awk '
  NR==1  {min=$0}
  NR==19 {p95=$0}
  END    {print "min="min"s  p95="p95"s  target=0.200s"}'
Целевой результат (NFR)
min=0.00Xs p95=0.XXXs target=0.200s ✓ p95 должен быть ниже 0.200s
