# Skill: tester (Flow Universe)

Цель: дать компактный чеклист для ручного и semi‑auto тестирования ключевых фич Flow Universe MVP.

## 1. Auth и сессии

- **Регистрация/логин**
  - Проверить логин/логаут/refresh для всех преднастроенных пользователей (admin/manager/dev/viewer).
  - Неверный пароль → корректная ошибка без утечки деталей.
  - По logout все защищённые страницы становятся недоступны (редирект на login).

- **Роли**
  - ADMIN видит и может управлять пользователями и проектами.
  - MANAGER видит проекты своей команды, может создавать/редактировать задачи и спринты.
  - USER видит только свои проекты/задачи и может менять статусы в рамках прав.
  - VIEWER только читает, не может изменять данные.

## 2. Projects & Issues

- CRUD проектов:
  - Создание проекта с ключом (`KEY`) → новые задачи получают ключ вида `KEY-1`, `KEY-2` и т.д.
  - Редактирование/архивация проекта доступны только ролям с правами.
- Иерархия задач:
  - EPIC → STORY → TASK → SUBTASK, отдельно BUG.
  - Проверить ограничения по родителям/детям.
- Статусы:
  - Базовый workflow: OPEN → IN_PROGRESS → REVIEW → DONE / CANCELLED.
  - Нельзя перескочить шаг, если это не разрешено бизнес‑логикой.

## 3. Sprint 2: Board, Sprints, Time, Comments

- **Kanban board**
  - Колонки соответствуют статусам.
  - Drag‑n‑drop перетаскивает задачу и меняет статус на backend.
  - Обновление страницы не ломает порядок и статусы.

- **Sprints**
  - Создание спринта в проекте, старт, закрытие.
  - Перемещение задач между бэклогом и спринтом (перед стартом и во время, если это поддерживается правилами).
  - При закрытии спринта задачи в незавершённых статусах корректно обрабатываются (возврат в бэклог/перенос).

- **Time tracking**
  - Старт/стоп таймера на задаче записывают лог в БД (проверить суммарное время).
  - Ручной ввод времени (валидные диапазоны, нельзя задать отрицательную длительность).
  - Нельзя иметь одновременно несколько активных таймеров для одного пользователя (если это требование есть в ТЗ/коде).

- **Comments**
  - Добавление, редактирование (если разрешено), удаление комментариев на задаче.
  - Права: свои комментарии vs чужие, роли с админ‑правами.
  - Отображение корректного порядка и времени.

## 4. Безопасность и негативные сценарии

- Доступ к API без токена → 401.
- Доступ к чужому проекту/задаче без прав → 403.
- Попытка изменить задачу в закрытом спринте (если запрещено) → предсказуемая ошибка.
- Все мутации должны попадать в AuditLog (выборочно проверять в БД или через admin‑интерфейс, если есть).

## 5. Нефункциональные проверки

- Основные страницы открываются < 2 секунд при нормальной нагрузке dev‑стенда.
- Частые операции (переключение статусов задач, работа с доской) не «залипают» и не приводят к рассинхрону UI ↔ backend.
