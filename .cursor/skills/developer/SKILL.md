# Skill: developer (TaskTime)

## 1. Когда включать этот скилл

Используй этот скилл, когда нужно:
- Добавить или изменить функциональность в TaskTime MVP.
- Проектировать API/модуль по паттернам текущей архитектуры.
- Править код, затрагивающий несколько слоёв (frontend + backend + БД).

Всегда исходи из того, что **старый vanilla-прототип мёртв**. Работай только с текущим стеком.

## 2. Архитектура и стек (обязательно учитывать)

- **Язык:** TypeScript 5.x.
- **Backend:** Node.js 20, Express 4, модульный монолит.
- **ORM:** Prisma 6 + PostgreSQL 16.
- **Frontend:** React 18, Vite 6, Ant Design 5, Zustand.
- **Кэш:** Redis 7 (использовать, но не превращать в ключевой источник правды).
- **Тесты:** Vitest + Supertest.
- **Модули backend:** `auth`, `users`, `projects`, `issues`, `comments`, `boards`, `sprints`, `time`, `teams`, `audit`, `admin`.
- Каждый модуль: `router → service → prisma` (никаких прямых SQL, никаких cross‑module repository imports).

## 3. Backend-паттерны

- **Модульность**
  - Любая новая бизнес‑логика должна приземляться в существующий или новый модуль.
  - Структура: `module.router.ts` (маршруты) → `module.service.ts` (бизнес‑логика) → `prisma`‑запросы.
  - Не импортировать сервисы/репозитории из других модулей; обмен данными — через чёткие API (router) или общие утилиты.

- **DTO и валидация**
  - Входные данные валидировать через Zod‑схемы.
  - Вынести DTO/схемы в отдельные файлы модуля (`dto.ts`/`schema.ts`).
  - Не доверять клиенту: повторять критичные проверки на backend (статусы, права, границы).

- **Ошибки и аудит**
  - Любые ожидаемые ошибки кидать через унифицированные error‑классы, чтобы глобальный middleware мог отформатировать ответ.
  - Все изменения важных сущностей (issue, project, user, sprint, time log) должны попадать в `AuditLog` (через middleware или явные вызовы).

## 4. Auth и RBAC

- **Auth**
  - Использовать JWT‑access + refresh‑token (storage по текущей архитектуре проекта).
  - Любые новые endpoint’ы, требующие авторизации, должны:
    - Подключать auth‑middleware.
    - Явно проверять роль (ADMIN/MANAGER/USER/VIEWER) и, где нужно, владельца/участника.

- **RBAC**
  - Глобальные роли: ADMIN, MANAGER, USER, VIEWER.
  - Перед любой чувствительной операцией:
    - Проверить, что пользователь **не заблокирован** (`is_blocked`).
    - Проверить, что роль и контекст (проект/команда) разрешают действие.
  - Никогда не полагаться только на проверку на фронте.

## 5. Frontend-паттерны

- **React + Ant Design**
  - Страницы строить как React‑SPA на Ant Design 5 (Layout, Menu, Table, Form, Modal, Drawer и т.д.).
  - Не использовать ручную DOM‑манипуляцию (`document.querySelector`, inline‑скрипты и т.п.).
  - Состояние формы и UI — через React state / hooks, глобальное состояние — через Zustand.

- **Состояние и запросы**
  - Общие сущности (текущий пользователь, выбранный проект, фильтры доски) хранить в Zustand‑store.
  - Запросы к backend оформлять в отдельном слое API‑клиента (типы ответов выводить из backend DTO, не лепить `any`).
  - Обязательно обрабатывать состояния: loading, error, empty.

- **Навигация**
  - Использовать React Router (страницы: login, dashboard, projects, project detail, issue detail, board и т.д.).
  - Никаких self‑made `pageMap` и ручного управления `history` как в старом прототипе.

## 6. Работа с данными и миграциями

- Все изменения схемы БД — через Prisma Migrate.
- Сначала проектируй модель в терминах домена (User, Project, Issue, Sprint, Comment, TimeLog, AuditLog) → затем отражай в Prisma schema.
- Seed‑данные и миграции должны быть идемпотентными (безопасно прогоняться повторно в dev/staging).

## 7. Тесты и качество

- Для нового функционала по возможности добавлять:
  - Unit/интеграционные тесты Vitest + Supertest для backend endpoint’ов.
  - Простые тесты ключевых React‑компонентов (формы, критичные workflow).
- Не ломать существующие тесты; при необходимости обновлять снапшоты и DTO строго по изменённой контрактной логике.
