# Специалист по информационной безопасности

## Зона ответственности

- Защита информации: разграничение доступа, аутентификация, учёт; шифрование и целостность данных при необходимости.
- Безопасная конфигурация (сервисы, сеть, ОС); реагирование на уязвимости и зависимости.
- Учёт отраслевых и контрактных требований по ИБ без автоматического применения режима КИИ.

## Обязательный контекст

Учитывать контекст импортозамещения: [ru-compliance-context](../ru-compliance-context/SKILL.md). Система не ЗоКИИ; структура требований по импортозамещению; допущения и легальные варианты использования иностранного ПО и оборудования. Рекомендации по ИБ не должны по умолчанию исходить из норм КИИ; при необходимости — явно указывать допущения и рекомендовать согласование с юристом/комплаенсом.

## Выход

Рекомендации по ИБ, чек-листы, допущения по используемым средствам защиты; при необходимости — отсылка к комплаенсу.

## Известные грабли: браузерные куки и аутентификация

### HttpOnly-кука, установленная через fetch(), может не прийти в навигационный запрос

**Проблема.** Схема «сервер проверяет куку при GET /app и редиректит на `/?blocked=1`, если куки нет» ломается в Safari (ITP) и ряде версий Chrome:
- куку устанавливает `fetch()` → `Set-Cookie`
- браузер немедленно делает `window.location.href = '/app'`
- навигационный запрос уходит **без куки** → сервер считает пользователя неавторизованным

**Правило.** Серверный cookie-gate на раздаче статического HTML — **ложная защита**. Реальная защита — на API-эндпоинтах через `authMiddleware`. Для SPA: отдавать HTML безусловно, auth-проверку делать на клиенте (localStorage → `/api/auth/me` → редирект).

**Что проверять при код-ревью:**
- Нет ли `res.redirect` на основе куки при отдаче HTML-страниц?
- Все ли мутирующие API-эндпоинты защищены `authMiddleware`?
- `sameSite`, `secure`, `httpOnly` — выставлены корректно для среды (dev/prod)?

### SameSite и Secure: типичные ошибки

| Ситуация | Проблема |
|---|---|
| `secure: true` + HTTP (без HTTPS) | Браузер тихо выбрасывает куку — авторизация ломается |
| `sameSite: 'Strict'` + переход с внешнего домена | Кука не отправляется (например, при SSO или OAuth-редиректе) |
| `sameSite: 'None'` без `secure: true` | Современные браузеры игнорируют куку |

**Для TaskTime:** `secure` включается только при `NODE_ENV=production`; убедиться, что в production всегда HTTPS.

### Пасхалка как security UX

Страница-заглушка (`?blocked=1`) — намеренный элемент: показывает, что `/app` недоступен без авторизации. Срабатывать должна **только на клиенте** (JS в `app.html` при провале `/api/auth/me`), не на сервере при раздаче статики.

## RBAC: блокировка пользователей

### Флаг is_blocked: принцип работы и ограничения

Поле `is_blocked BOOLEAN DEFAULT FALSE` в таблице `users` проверяется **только при логине** — не в каждом запросе через JWT.

**Последствие:** уже залогиненный пользователь (с выданным JWT) продолжит работать до истечения токена (7 дней по умолчанию). Это приемлемо для MVP, но не для high-security.

**Что проверять при ревью:**
- Блокировка логина: `SELECT is_blocked` перед выдачей токена
- Audit-запись при попытке входа заблокированным (`level: 'warning'`)
- Нельзя заблокировать самого себя через admin panel
- Нельзя заблокировать super-admin если ты admin

**Для полной блокировки (сессионная инвалидация):** нужна таблица revoked_tokens или sessions — это tech debt.

### Защита admin endpoints: два middleware подряд

```
authMiddleware → проверка JWT токена (кто ты)
adminMiddleware → проверка роли (имеешь ли ты право)
```

Оба middleware обязательны на каждом `/api/admin/*`. Без `authMiddleware` перед `adminMiddleware` — дыра: `req.user` будет `undefined`.

### Проверка роли на клиенте — не замена серверной

Клиентская проверка роли (скрыть кнопку, показать заглушку) — UX, не безопасность. Сервер должен отвергать любые запросы без правильной роли, даже если UI «скрыл» возможность.
